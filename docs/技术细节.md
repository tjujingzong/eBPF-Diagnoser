# eBPF Diagnoser - 技术细节

## 支持的异常场景

| 场景 | 检测能力 | 根因定位 |
|------|---------|---------|
| CPU异常占用 | CPU使用率、上下文切换、调度延迟 | CPU密集计算/线程竞争/busy loop |
| I/O延迟抖动 | IOPS、P99时延、队列深度、热点设备 | 队列过深/缓存失效/热点文件 |
| 内存抖动/OOM | 内存使用率、page fault、kswapd活跃度 | 进程泄漏/系统内存不足/OOM风险 |
| 锁竞争 | futex等待时间、争用率、热点调用栈 | 临界区过大/锁粒度过粗 |
| 系统调用异常 | syscall频率、耗时分布、慢syscall | 轮询/阻塞型调用/错误率高 |

## 架构设计

核心架构分为四层：
1. **BPF探针层** (C语言): 5个独立BPF程序，通过tracepoint挂载到内核，零拷贝采集指标
2. **加载器层** (C bpf_loader): 长驻进程，通过stdin/stdout JSON协议提供BPF程序加载、挂载和map读取
3. **分析层** (Python): 指标聚合(滑动窗口+动态基线) + 规则引擎(可配阈值) + 跨场景关联分析
4. **输出层** (Python): JSON/YAML/Markdown/终端表格四种格式
5. **LLM增强层** (可选): 基于云端LLM的智能分析、修复建议、交互问答、日志分析

## 压测验证

```bash
# CPU压测 + 诊断
bash scripts/stress_cpu.sh &
sudo ebpf-diagnoser run --probe cpu --duration 60

# 全场景组合压测
bash scripts/stress_composite.sh &
sudo ebpf-diagnoser run --probe all --duration 120 --output all

# 运行内置功能测试 (自动施加压测并验证异常检测)
sudo ebpf-diagnoser test

# 性能开销基准测试
sudo bash scripts/benchmark_overhead.sh
```

## 性能开销目标

| 指标 | 目标 |
|------|------|
| CPU开销 | < 2% 单核 |
| 内存开销 | < 100MB |
| I/O P99时延影响 | < 5% |
| 系统吞吐下降 | < 3% |

## 限制说明

- 需要root权限或CAP_BPF/CAP_PERFMON/CAP_SYS_ADMIN能力
- 当前仅支持x86_64架构(ARM64可通过修改BPF头文件支持)
- BPF探针使用tracepoint挂载，依赖内核调度/块设备/内存管理等子系统的tracepoint接口
- 动态基线需要30个样本的预热期，前30秒采集使用静态阈值
- 系统调用探针在全采样模式下(sample_rate=1)对高频syscall场景有一定开销，建议生产环境设置采样率>1
- 异常抑制窗口为30秒，同类异常在30秒内不重复报告

## LLM智能分析功能

### 支持的LLM提供商

| 提供商 | 模型示例 | 配置值 |
|--------|----------|--------|
| OpenAI | gpt-4o-mini, gpt-4o | `provider: openai` |
| Anthropic | claude-3-haiku, claude-3-sonnet | `provider: anthropic` |

### 功能列表

| 命令 | 功能 | 说明 |
|------|------|------|
| `ebpf-diagnoser analyze` | 智能分析报告 | 基于诊断数据生成深度分析 |
| `ebpf-diagnoser remediate` | 自动修复建议 | 生成可执行的修复命令 |
| `ebpf-diagnoser chat` | 交互式问答 | 与LLM对话分析系统问题 |
| `ebpf-diagnoser log-analyze` | 日志智能分析 | 结合日志与eBPF数据关联分析 |

### 配置方式

**环境变量（推荐）：**
```bash
export EBPF_DIAGNOSER_API_KEY="your-api-key"
export EBPF_DIAGNOSER_LLM_PROVIDER="openai"  # 可选
export EBPF_DIAGNOSER_LLM_MODEL="gpt-4o-mini"  # 可选
```

**配置文件：**
```yaml
llm:
  provider: openai
  api_key: your-api-key
  model: gpt-4o-mini
  temperature: 0.7
  max_tokens: 4096
```

### 成本控制

- 默认使用gpt-4o-mini（成本低）
- 输入token超限时自动截断
- 支持结果缓存避免重复调用
- LLM不可用时回退到规则引擎分析
