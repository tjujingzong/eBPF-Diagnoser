# eBPF Diagnoser 设计说明

## 1. 整体架构

eBPF Diagnoser 采用四层分层架构，将 BPF 数据采集与 Python 分析逻辑解耦：

```
┌─────────────────────────────────────────────────────────┐
│                    输出层 (Python)                       │
│    JSON / YAML / Markdown / 终端表格                     │
├─────────────────────────────────────────────────────────┤
│                   分析层 (Python)                        │
│    规则引擎 ─→ 异常检测 ─→ 跨场景关联分析                │
├─────────────────────────────────────────────────────────┤
│                  采集层 (Python)                         │
│    指标聚合(滑动窗口) ─→ 动态基线(3-sigma)               │
├─────────────────────────────────────────────────────────┤
│              BPF探针层 (C + libbpf)                      │
│    cpu_probe │ io_probe │ mem_probe │ lock_probe │ syscall_probe │
│    ↕ JSON协议 (stdin/stdout)                             │
│              bpf_loader (C长驻进程)                      │
└─────────────────────────────────────────────────────────┘
```

## 2. BPF 探针设计

### 2.1 技术方案选择

采用 **libbpf + CO-RE** (Compile Once, Run Everywhere) 方案：

- **编译时**: 使用 clang 将 BPF C 源码编译为 .o 文件，BTF 信息嵌入其中
- **运行时**: C bpf_loader 进程通过 libbpf 加载 .o 文件，自动完成 CO-RE 重定位
- **优势**: 不依赖 BCC 运行时编译，无 LLVM 依赖链；预编译的 .o 可跨内核版本运行

### 2.2 五个探针的 tracepoint 挂载

| 探针 | tracepoint | 采集指标 |
|------|-----------|---------|
| CPU探针 | `sched:sched_switch`, `sched:sched_wakeup` | CPU使用率、调度延迟、上下文切换、运行队列 |
| I/O探针 | `block:block_rq_issue`, `block:block_rq_complete` | IOPS、P99延迟、队列深度、热点设备 |
| 内存探针 | `vmscan:mm_vmscan_direct_reclaim_begin/end`, `vmscan:mm_vmscan_kswapd_wake/sleep` | 内存使用率、page fault、reclaim/kswapd活跃度 |
| 锁探针 | `sched:sched_process_futex_wait` (内核5.x+) 或 `syscalls:sys_enter_futex/sys_exit_futex` | futex等待时间、热点进程、争用率 |
| 系统调用探针 | `raw_syscalls:sys_enter`, `raw_syscalls:sys_exit` | syscall频率、耗时分布、慢syscall |

### 2.3 BPF Map 设计

每个探针使用以下类型的 BPF Map：

- **ARRAY Map**: 全局计数器（如 `total_syscalls`、`sample_rate_map`），单元素高性能读写
- **HASH Map**: 按key聚合的统计信息（如 `syscall_stats` 按syscall号、`io_stats` 按设备号），支持动态key
- **PERCPU_ARRAY/PERCPU_HASH**: 在需要避免原子操作的高频路径使用

### 2.4 采样率控制

通过 `sample_rate` 参数控制采样频率（1/N）：

- `sample_rate = 1`: 全采样，每个事件都记录
- `sample_rate = N`: 每 N 个事件记录 1 个

采样计数器存储在 BPF ARRAY Map 中，通过 `__sync_fetch_and_add` 原子操作实现。

## 3. bpf_loader 通信协议

### 3.1 架构

`bpf_loader` 是一个 C 语言编写的长驻进程，通过 stdin/stdout 与 Python 端进行 newline-delimited JSON 通信。

```
Python (src/probes/bpf_loader.py)
    ↓ stdin (JSON)
bpf_loader (C进程)
    ↑ stdout (JSON)
```

### 3.2 支持的命令

| 命令 | 说明 | 参数 |
|------|------|------|
| `LOAD` | 加载BPF .o文件 | `obj_path`, `sample_rate` |
| `ATTACH` | 挂载BPF程序到tracepoint | `obj_index` |
| `DETACH` | 卸载BPF程序 | `obj_index` |
| `READ_MAP_ARRAY` | 读取ARRAY map | `obj_index`, `map`, `index` |
| `READ_MAP_HASH` | 遍历HASH map | `obj_index`, `map`, `max_entries` |
| `READ_STACK` | 读取栈追踪 | `obj_index`, `map`, `stack_id` |
| `RESOLVE_KSYM` | 解析内核符号 | `addrs` (地址数组) |
| `WRITE_MAP_ARRAY` | 写入ARRAY map | `obj_index`, `map`, `index`, `value` |
| `QUIT` | 退出loader进程 | 无 |

### 3.3 BTF 序列化

bpf_loader 利用 BPF 对象的 BTF (BPF Type Format) 信息自动将 map 数据序列化为 JSON：

- **整数类型** (u8/u16/u32/u64): 直接输出数值
- **结构体**: 递归序列化为 JSON 对象，字段名来自 BTF
- **字符数组**: 序列化为 JSON 字符串

## 4. 分析引擎设计

### 4.1 指标聚合 (MetricsAggregator)

- **滑动窗口** (MetricsWindow): 保留最近 60 秒的指标快照
- **动态基线** (DynamicBaseline): 基于历史数据计算均值和标准差
  - 预热期: 前 30 个样本使用静态阈值
  - 异常判定: 超出 3-sigma 范围视为异常
  - 历史容量: 300 个样本 (5分钟@1s间隔)

### 4.2 规则引擎 (RuleEngine)

采用声明式规则定义，每条规则包含：

```python
{
    "name": "规则名称",
    "conditions": [
        {
            "metric": "cpu.global.cpu_usage_percent",  # 指标路径(支持嵌套)
            "op": ">",                                  # 比较操作
            "value": 90,                                # 默认阈值
            "threshold_key": "cpu_usage_high",          # 配置阈值键名
            "threshold_factor": 1.0,                    # 阈值系数
            "use_baseline": False,                      # 是否使用动态基线
            "description": "条件描述"
        }
    ],
    "anomaly_type": "cpu_anomaly",
    "severity": "high",
    "root_cause": { "description": "...", "category": "...", "confidence": 0.85 },
    "recommendations": ["..."]
}
```

**阈值解析优先级**:
1. `threshold_key` 引用的配置阈值 × `threshold_factor`
2. 条件中的 `value` 硬编码值
3. 命令行 `--threshold` 覆盖

**动态基线模式**: 当 `use_baseline=True` 时，阈值变为 `baseline_mean + value * baseline_stdev`，实现自适应检测。

### 4.3 跨场景关联分析 (CorrelationEngine)

支持以下跨场景关联规则：

| 组合 | 关联条件 | 增强效果 |
|------|---------|---------|
| CPU + I/O | iowait > 20% | I/O异常根因修正为"I/O导致CPU等待" |
| 内存 + I/O | pswpin+pswpout > 10/s | I/O异常根因修正为"内存压力导致换页I/O" |
| 锁 + CPU | ctx_switch > 20000/s | CPU异常证据链增加锁竞争关联 |
| 内存 + 锁 | pgfault > 50000/s | 锁异常根因修正为"内存分配锁争用" |
| CPU + Syscall | total_syscalls > 100000 | CPU异常根因修正为"高频syscall开销" |

### 4.4 异常抑制

- 同类异常在 30 秒窗口内不重复报告
- 每次报告更新时间窗口 (start/end)

## 5. 配置系统

采用三层优先级配置：

```
命令行参数 (--threshold key=value)   [最高优先级]
    ↓
配置文件 (config/default.yaml)       [中等优先级]
    ↓
代码默认值 (src/config.py dataclass)  [最低优先级]
```

### 5.1 可配置阈值

| 阈值键名 | 说明 | 默认值 |
|---------|------|--------|
| `cpu_usage_high` | CPU使用率告警(%) | 90 |
| `cpu_usage_warn` | CPU使用率警告(%) | 70 |
| `sched_delay_high` | 调度延迟告警(ms) | 50 |
| `io_p99_high` | I/O P99延迟告警(ms) | 50 |
| `mem_available_low` | 可用内存告警(%) | 10 |
| `lock_wait_high` | 锁等待告警(ms) | 10 |
| `syscall_freq_high` | syscall频率告警(倍数) | 5 |

### 5.2 探针配置

| 配置项 | 说明 | 默认值 |
|--------|------|--------|
| `cpu_sample_rate` | CPU采样率(1/N) | 1 |
| `io_sample_rate` | I/O采样率 | 1 |
| `mem_sample_rate` | 内存采样率 | 1 |
| `lock_sample_rate` | 锁采样率 | 10 |
| `syscall_sample_rate` | syscall采样率 | 1 |
| `enable_stack_trace` | 启用栈回溯 | false |
| `pid_filter` | PID过滤 | [] (全部) |
| `comm_filter` | 进程名过滤 | [] (全部) |

## 6. 输出格式

### 6.1 结构化输出字段

所有异常输出至少包含以下字段：

| 字段 | 说明 |
|------|------|
| `type` | 异常类型 (cpu_anomaly/io_anomaly/memory_anomaly/lock_anomaly/syscall_anomaly) |
| `severity` | 严重程度 (low/medium/high/critical) |
| `confidence` | 置信度 (0-1) |
| `time_window` | 异常时间窗口 (start/end) |
| `affected_objects` | 受影响对象 (进程/设备/系统调用) |
| `key_metrics` | 关键指标 (指标名→值) |
| `evidence_chain` | 证据链 (步骤列表，含指标、值、基线) |
| `root_cause` | 疑似根因 (描述、类别、置信度、推理过程) |
| `recommendations` | 建议列表 |

### 6.2 支持的输出格式

- **JSON**: 完整结构化报告，适合机器解析和API集成
- **YAML**: 可读性更好的结构化报告
- **Markdown**: 人类可读的诊断报告，含表格和emoji标记
- **终端表格**: 彩色终端输出，适合交互式使用

## 7. 扩展设计

### 7.1 添加新探针

1. 编写 BPF C 程序 (`bpf/new_probe.bpf.c`)
2. 在 Makefile 中添加编译规则 (自动发现)
3. 实现 Python 探针类 (继承 `BaseProbe`)
4. 在 `src/probes/__init__.py` 的 `ProbeManager` 中注册
5. 在 `src/analyzer/rules.py` 中添加检测规则

### 7.2 添加新规则

在 `src/analyzer/rules.py` 的 `RuleEngine._load_default_rules()` 中添加规则，或在 `config/` 目录下创建 YAML 规则文件。

### 7.3 添加新输出格式

在 `src/output/formatter.py` 中实现新的 `format_xxx()` 函数，并在 `OutputFormatter` 类中注册。
